⚠ Aviso honesto de escopo

O Nemesis Defender é um sistema de enforcement robusto e ativo, em camadas independentes, validado em produção e contra agentes adversariais reais. Como qualquer sistema de segurança sério, ele não promete cobrir todo vetor possível declara o que faz, contra qual modelo de ameaça, e onde estão seus limites. Honestidade sobre escopo é uma escolha de engenharia, não uma fraqueza.

⚠️ LEIA ANTES DE INSTALAR · enforcement ativo e irreversível

O Nemesis Defender é um sistema de segurança que age, não pergunta. Não negocia e não abre concessões: a decisão é binária (0 ou 1).

O Nemesis age: ao detectar hostilidade confirmada num arquivo (credencial exposta, payload destrutivo, malware), ele move o arquivo para quarentena (.nemesis/quarantine/) e bloqueia a sessão até revisão humana. É enforcement determinístico, não um aviso. Com o modelo de severidade por corroboração, isso só ocorre quando a ameaça é confirmada por sinais de detecção independentes: código legítimo não é movido por engano. O conteúdo é preservado e você decide: restore (falso-positivo) ou purge.

Isto é intencional. O Nemesis intercepta operações destrutivas independentemente da origem, seja IA ou humano. No Linux com eBPF, o bloqueio vale inclusive para o operador no terminal (por exemplo, um rm -rf). Requer conhecimento técnico mínimo para configurar; não é plug-and-play.

Não é um linter genérico nem um scanner passivo. É um harness de enforcement em runtime para o contexto de desenvolvimento guiado por agentes de IA, onde um output aparentemente inócuo pode esconder manipulação de manifests, decode-then-exec de payloads codificados, esteganografia Unicode (CVE-2021-42574), prompt injection, exfiltração de credenciais e malware com time-gating. O Nemesis bloqueia essas operações no momento em que tentam executar.

O problema é real e atual: em 2025 foram publicados centenas de milhares de pacotes maliciosos no npm, e campanhas como Shai-Hulud comprometeram pacotes com bilhões de downloads semanais. O Nemesis Defender ataca a janela em que esse código tenta executar na máquina do desenvolvedor.

★ Isto já não existe nativamente?

O que as IDEs e TUIs já trazem e por que não resolve sozinho

Os primitivos já existem: hooks de ciclo de vida, deny-list e sandbox são recursos nativos das IDEs e agentes, e o Nemesis Defender os utiliza. Não inventei o hook nem o sandbox; consolidei essas peças num único framework de enforcement, uniforme entre IDEs, somado a uma camada de kernel que nenhuma delas oferece. Nenhuma ferramenta nativa entrega essa combinação completa.

E isto não é específico de uma ferramenta. Consultei a documentação oficial de cada agente: todos os principais convergiram para o mesmo primitivo um hook de ciclo de vida que intercepta a chamada antes de executar e pode bloquear. A tabela abaixo mostra o mecanismo de bloqueio de cada um, direto da doc oficial.

Agente / IDE Hook de bloqueio (doc oficial) Como bloqueia
Claude Code PreToolUse exit code 2, ou permissionDecision: deny via JSON. Só o exit 2 bloqueia; exit 1 é ignorado.
OpenAI Codex hooks + sandbox/approvals + execpolicy Hooks exigem trust-review por hash; múltiplos hooks do mesmo evento rodam concorrentes, então um não impede o outro. Sandbox workspace-write / approval policy.
Cursor (1.7+) beforeShellExecution, beforeReadFile, afterFileEdit Processo standalone, JSON via stdin/stdout, retorna permission allow/deny. Todos os hooks definidos rodam, sem precedência entre eles.
GitHub Copilot preToolUse (em .github/hooks/) O hook mais poderoso: aprova ou nega a execução de ferramenta. Roda síncrono e bloqueia o agente; doc recomenda manter abaixo de 5s.
VS Code (agent, preview) 8 eventos de hook em .github/hooks/ JSON, type command. A própria doc alerta: se o agente pode editar o script do hook, ele pode reescrevê-lo durante a execução e rodar o que escreveu.
Devin (hoje Devin, da Cognition) Cascade local + Devin (cloud/terminal) Onde o Nemesis nasceu nativamente. Pós-aquisição (Devin 2.0, abr/2026), o Devin para Terminal roda com acesso completo à codebase, ampliando a superfície a proteger.

O padrão é claro: o primitivo é o mesmo em todos, e em todos ele é apenas o ponto de interceptação. Note também que "exit code 2" não é um mecanismo de segurança dedicado: na convenção POSIX, 0 é sucesso e qualquer não-zero é falha, e o 2 historicamente sinaliza uso indevido de built-in do shell. Os agentes reaproveitaram um código de erro genérico do shell como sinal de "bloqueie". Toda a lógica que decide o que é perigoso continua sendo escrita por você.

Recurso nativo O que de fato faz Onde para (evidência)
Hooks (Claude Code PreToolUse, Cursor beforeShellExecution, Copilot) Interceptam a chamada de ferramenta antes de executar e podem bloquear (exit code 2). É só o ponto de interceptação você tem que escrever toda a lógica de enforcement. E é por-IDE, com payloads incompatíveis entre si (snake_case vs camelCase). Não vem nada pronto.
Deny-list / permissions (settings.json allow/deny) Bloqueia comandos e paths por nome para as ferramentas internas do agente; deny vence allow. Pela própria doc da Anthropic: deny-rules só bloqueiam as ferramentas internas Read(./.env) é barrado, mas cat .env no bash passa por cima. Não inspeciona conteúdo.
Sandbox / isolamento (/sandbox bubblewrap·Seatbelt, devcontainer/Docker) Isolamento de filesystem e rede no nível do SO. Reduz o raio de dano (blast radius). É o recurso nativo mais forte. Só cobre o Bash, não as ferramentas Read/Edit (issue #26616 do Claude Code). E ele contém, não detecta: um package.json envenenado ou uma cadeia de exfiltração passam "dentro" do sandbox. Enfraquece em modo nested e é por-IDE.
Rules / skills / workflows (.cursorrules, CLAUDE.md, etc.) Dão contexto e diretrizes ao modelo sobre o que fazer e o que evitar. São advisory, não enforcement. O modelo é um motor de previsão, não um aplicador de política ignora, reinterpreta ou sobrescreve (relatos no fórum do Cursor; análise da Knostic). O código vazado do Claude Code mostrou que até o agente da Anthropic trata constraints como dicas.
eBPF / LSM no kernel (KubeArmor, Tetragon, Falco) Enforcement de syscall no kernel via BPF LSM (5.7+), retornando -EPERM a defesa com mais "dentes", que vale inclusive contra o humano no terminal. Opera sobre estado já residente no kernel, depois do dado copiado do user space, então evita as races TOCTOU de quem só inspeciona texto em userspace (doc oficial do kernel). Existe e é maduro mas no mundo cloud/Kubernetes. Não é entregue como binário local, por-projeto, acoplado ao ciclo de um coding agent. É exatamente a camada eBPF do Nemesis, trazida para a máquina do dev.
O que isso prova na prática

A diferença entre "configurar uma regra" e "forçar uma regra" não é teórica. Em julho de 2025, o agente do Replit deletou um banco de produção durante um code freeze ativo, apesar de instruções repetidas porque, segundo a análise do incidente, as instruções não eram tecnicamente forçadas: não havia um gate que bloqueasse a ação (AI Incident Database #1152). Na mesma linha estão o Gemini CLI apagando arquivos por interpretar mal um comando e o Claude Code rodando um terraform destroy acidental. Settings, rules e até sandbox parcial não impediram nenhum deles. É essa lacuna entre o que o modelo promete e o que ele de fato executa que um enforcement determinístico fecha.

Resumindo: a deny-list nativa não pega cat .env; o sandbox nativo não pega Read/Edit nem um pacote envenenado; as rules não pegam nada porque são advisory; e o eBPF de kernel mora na nuvem, não na sua máquina. Cada peça resolve um pedaço no máximo alguns comandos destrutivos, e não todos. O Nemesis Defender junta todas as peças num único framework, com deny-list + scanner de conteúdo (destrutivo e malicioso) + eBPF no kernel como última camada para usuários Linux, igual em Devin, Cursor, Codex, Claude Code e VS Code. Construi esse framework que resolveu o meu problema e abri sob AGPL-3.0 para quem quiser usar.

02 Gênese

Nasceu da dor, não da prancheta

O Nemesis Defender não foi projetado num quadro de segurança. Cresceu de uma dor concreta e repetida dentro de projetos reais de produção e evoluiu em três fases.

Fase 1 · regras em markdown

Convenções escritas na IDE

No início eram só regras em texto, tentando conter os mesmos erros que se repetiam: hooks condicionais, setState em useEffect, any, tipagens duplicadas, CSS fora do tailwind.config, lógica de negócio em arquivos de UI. A lição: regra escrita é input de contexto o modelo lê, entende, e ainda assim executa o padrão neural de "resolver rápido". Instrução não é enforcement.

Fase 2 · Node/TypeScript

O linter automático

As convenções viraram hooks que rodam e bloqueiam de fato. O primeiro salto foi um linter automático: em vez de "por favor não use any", o sistema passou a barrar o anti-pattern antes de entrar no repositório. Determinístico, não negociável.

Fase 3 · Rust + segurança

Do linter ao escudo de kernel

Reescrito em Rust, com o foco virando 100% segurança: deny-list de comandos, scanner de conteúdo, detecção de supply-chain e a camada eBPF no kernel. A mesma filosofia que barrava any passou a barrar rm -rf destrutivo e exfiltração de credenciais.

A linha que conecta as três fases: as regras são rígidas porque a dor das violações foi real e recorrente. Só o enforcement que roda fora da inferência (lint-time, PreToolUse, kernel) fecha o gap entre regra e comportamento.

03 Por que existe

Impedir o comando destrutivo, antes dele rodar

A função mais importante é cotidiana: impedir que um agente execute, sem intenção, um comando que destrói o projeto rm -rf no lugar errado, git reset --hard que apaga trabalho não commitado, sobrescrita de config.

Quem programa com agentes LLM sabe que isso não é hipótese: o modelo, tentando "ajudar", roda um comando destrutivo porque interpretou mal o contexto. O Nemesis Defender intercepta antes da execução e exige confirmação humana. Vale para o erro involuntário (o caso comum) e para o código malicioso.

Modelos operam por inferência probabilística pedir "não rode comandos destrutivos" é instrução que o modelo pode ignorar, errar ou rotear ao redor. O enforcement determinístico antes da execução é categoricamente diferente: não importa se o modelo foi enganado ou apenas errou, a camada bloqueia. Ferramentas reativas (linters, CI/CD, SAST) detectam depois; o Nemesis Defender bloqueia antes. Complementar a elas não substituto.

04 Linha de autoridade

O modelo opera; ele não destrói

O coração do design é uma divisão clara do que a IA pode e não pode fazer:

  • A IA usa as ferramentas nativas de escrita e edição da IDE mas só dentro do scaffold do projeto.
  • Pode ler alguns arquivos sensíveis quando o trabalho exige, mas não editar nem apagar nenhum deles.
  • Exclusão, sobrescrita fora do escopo, reset, comando destrutivo: prerrogativa exclusivamente humana, não delegável.

Essa fronteira não depende da boa vontade do modelo é imposta pelas camadas abaixo. O "zero incidente" em produção não vem do modelo "se comportar", mas da arquitetura nunca lhe dar essa autonomia.

05 Arquitetura

Tudo parte do Pretool

O Pretool Hook é o ponto de entrada de todo o sistema sem ele, o Nemesis não funciona. A trilha de segurança (Defender) roda acoplada ao pretool: é acionada porque o pretool a invoca na mesma interceptação. A camada eBPF é a única que opera fora dessa linha no kernel, como reforço adicional para Linux.

01

Pretool / Posttool Hook base

Win · macOS · Linux

​Origem de tudo: sem pretool, o Nemesis não funciona. Intercepta comandos antes de Bash.run() (PreToolUse), valida contra deny-list.json e escopo hard-gate: sem aprovação humana, exit code 2. É ele que aciona as trilhas de Defender. Um hook PostToolUse roda depois (nunca bloqueia) e audita o output em nemesis-violations.log. E um fallback fail-closed: se o binário não existe, bloqueia tudo em vez de liberar.

02

Nemesis Defender junto com o pretool

Win · macOS · Linux

Acionado pelo pretool na interceptação. Scanner de conteúdo com 6 layers AST (tree-sitter) → byte → regex → denylist → entropia → decoder recursivo (máx. 3 níveis) e visitors cobrindo os vetores catalogados, incluindo taint-tracking de fonte→sink, exfil chains e poisoning de configs de IDE.

03

eBPF Kernel LSM adicional

Linux apenas

Camada adicional e independente do pretool, só para Linux. Vive no kernel e segura as pontas contra comandos destrutivos (e alguns malicious): se o pretool falhar ou for contornado e o modelo tentar destruir algo, o kernel ainda barra. Bloqueia syscalls perigosas (mount, ptrace, kexec_load, delete_module → EPERM); tem deny-list própria, atrelada à arquitetura (no Cargo.toml do módulo). Faz também controle de egress (lsm/socket_connect): nega conexões de saída para destinos fora de uma allowlist CIDR:porta (cgroup-scoped, opt-in via egress.toml), barrando exfiltração/C2 mesmo que um payload consiga rodar. Em produção (nemesis-violations.log) registra bloqueios de rm, shred, dd, kill, mount, nc e runtimes (python3, perl). Em macOS/Windows não existe ali a defesa fica nas trilhas do pretool.

Existe ainda uma camada adicional de qualidade de código presente no projeto, hoje silenciada: está em desenvolvimento, ajustando falsos-positivos antes de validar sua usabilidade como proteção.

A deny-list do Defender contém centenas de patterns em dezenas de categorias, e foi ampliada na v2.0 com vetores destilados do estado da arte (exfil surface estendida, taint-tracking, poisoning de config de IDE, typosquatting). Patterns são, por natureza, uma lista do que já foi antecipado aumentam o custo de um ataque conhecido, mas não cobrem o desconhecido.

Severidade por corroboração: quarentena só quando confirmado

O endurecimento mais importante da v2.x: o Iron Dome só age (move para quarentena, não deleta) quando a hostilidade é confirmada. Sinais de alta confiança (denylist curada, decode→exec, cadeia de exfil, reverse shell socket+exec, bypass do próprio Nemesis) bloqueiam sozinhos; sinais heurísticos (substring/padrão) exigem 2 métodos de detecção independentes concordando antes de deletar caso contrário ficam em Suspicious (registrado, mantido). A detecção de execução foi ainda estendida a múltiplos runtimes (reverse shell e exec ofuscado em Ruby/PHP/Go/Perl/Java/Lua, não só JS/Python/Bash), e a proteção de paths foi endurecida contra ofuscação (glob em diretório, cd para dir protegido, indireção por variável/$(<file)).

07 Customização

Tudo é configurável e só por humano

Princípio central: não há regra hardcoded. Os hooks são agnósticos e canônicos se um padrão está na deny-list, é bloqueado; se não está, passa. Não existe lógica de decisão escondida no código do hook.

  • Qualquer vetor, regex ou regra pode ser desativado, relaxado ou customizado editando os arquivos de deny-list.
  • Ampliar a cobertura (novo vetor) é adicionar uma entrada não recompilar lógica.
  • Apenas um humano faz essas mudanças. O modelo não altera a própria gaiola: os arquivos de config estão sob absolute_block.

Duas exceções exigem conhecimento mais profundo: a camada eBPF tem sua lista de syscalls atrelada à arquitetura (no Cargo.toml), e os visitors do Defender são código Rust de análise ampliá-los exige Rust e segurança. As deny-lists, por outro lado, são simples de ajustar.

08 Controle de acesso

Quem pode ler/editar o quê

O arquivo denylist-folder-files.json define, sob controle exclusivamente humano, o que o agente pode tocar. É aqui que a linha de autoridade vira prática, em dois níveis:

  • absolute_block bloqueio total (leitura + escrita + deleção): .env, .ssh/id_rsa, .bashrc/.zshrc, os settings/hooks de cada IDE (.claude/, .cursor/, .devin/) e o próprio .nemesis/.
  • write_block leitura permitida, escrita/edição bloqueada: package.json, next.config.js, eslint.config.mjs, .gitignore e os logs.
  • allowed_exceptions o scaffold liberado (ex.: /src/), onde o agente escreve e edita à vontade.
Bypass é humano

Só um humano comuta essas permissões. Precisa relaxar uma restrição para manutenção? O bypass é humano e explícito o agente nunca promove a si mesmo. E, independente de qualquer permissão de leitura/escrita, comando destrutivo permanece sempre proibido para a IA.

10 Validação

Três tipos de evidência

Da mais forte para a mais fraca. A ordem é deliberada: a evidência que mais importa é a que mais se parece com o uso real.

1 · Track record em produção

O Nemesis Defender nasceu em projetos reais de produção (que não podem ser divulgados) e roda há ~1 ano e meio em desenvolvimento ativo, com agentes LLM operando sobre o código. Nesse período, nenhum agente executou um comando destrutivo nem acessou arquivo sensível para edição/exclusão sem permissão. Ele resolveu e resolve um problema real: a proteção foi corroborada por pentests estáticos contra os próprios binários (resultados satisfatórios) e pentests reais executados por agentes LLM que provam que modelos podem ser manipulados a executar atividade hostil, e que mesmo assim o enforcement segura. Não é 100%: gaps abertos são vetores ainda não contemplados, e os conhecidos ficam como pontos de melhoria. Mas a proteção mais importante para qualquer dev impedir que o ambiente seja destruído por comandos destrutivos é a função-núcleo e a mais validada.

2 · Execução adversarial por agente real

Em sessões registradas (Devin no Linux e macOS, Cursor, Claude Code), agentes receberam suítes de ataque disfarçadas de "scaffolding" e tentaram executá-las tarefa a tarefa. O Nemesis Defender bloqueou cada uma no ponto de execução (no Write, no comando aceito), com mensagens específicas. É o teste correto da tese: assume que o modelo pode ser enganado e mede se o enforcement segura. Segurou.

Mensagens de bloqueio padronizadas

Toda interceptação emite uma de seis mensagens fixas e categorizadas o agente (e o humano) sabe exatamente por que foi bloqueado. Cinco são da trilha de segurança (pretool/posttool/Defender/eBPF); a última é de qualidade (deny-list). Quando útil, a mensagem traz o alvo no fim (· arquivo ou · comando).

  • Comando: NEMESIS SEC - COMANDO NAO PERMITIDO
  • Escrita em arquivo protegido: NEMESIS SEC - ACESSO NEGADO - ARQUIVO PROTEGIDO
  • Leitura de arquivo protegido: NEMESIS SEC - LEITURA NEGADA - ARQUIVO PROTEGIDO
  • Conteúdo malicioso: NEMESIS SEC - CONTEUDO MALICIOSO DETECTADO
  • Fora do escopo: NEMESIS SEC - ESCRITA FORA DO ESCOPO PERMITIDO
  • Qualidade: NEMESIS QUALITY - PADRAO DE CODIGO NAO PERMITIDO ANALISAR REGRAS!

3 · Suíte de vetores sintéticos

  • O que cobre: uma suíte de pentest com 184 testes em 26 módulos comandos destrutivos, exfiltração (incl. IMDS, shell history, wallets), reverse shells, persistência, taint-flow, poisoning de config de IDE, typosquatting mais um módulo dedicado a medir falso-positivo (conteúdo legítimo que deve passar).
  • O que NÃO prova: completude ou invulnerabilidade. Suíte do autor; passar nela é o piso esperado, e a calibração de falso-positivo é trabalho contínuo.

Os logs: as camadas operando

Tudo num ledger único e local .nemesis/logs/nemesis-violations.log uma linha padronizada por bloqueio (camada · mensagem · data · hora). nemesis-defender --log-stats resume por camada, por tipo e por dia. Registro 100% local: nada é enviado, exfiltrado ou telemetrado para fora da sua máquina sem servidor, sem coleta remota, sem "phone home". Os dados são seus, para auditar e validar a proteção.

  • eBPF (adicional, Linux): +2.000 bloqueios reais com "layer":"ebpf", permission_denied no kernel, independente de deny-list em userspace.
  • Defender (classificador, junto com o pretool): vereditos nomeados ([MALICIOUS] · reverse_shells) com evidência capturada e instrução de correção.
  • Correlação: detecta brute force ("N tentativas em 300s") e padrões compostos ("leitura de sensível + comando de rede"). Detecção comportamental, acima de regex simples.
Red-team contínuo: vetores encontrados e fechados

A evidência mais útil vem de um adversário real. Numa campanha de red-team registrada, um agente recebeu uma suíte de engenharia social reversa e tentou, ao vivo e com o pretool ativo, infiltrar, exfiltrar e neutralizar o próprio Nemesis. Ciclo após ciclo, vetores de evasão foram descobertos e fechados na origem, com cada correção validada (pentest 184/184, sem regressão): leitura de config protegida por glob/cd/indireção de variável; reverse shell e RCE ofuscado (Function-constructor) em múltiplos runtimes; e um typo que deixava o arquivo de instruções do agente gravável. Nenhum vetor de neutralização (desligar a proteção) foi encontrado a auto-proteção segurou cada tentativa de matar o daemon, sobrescrever o binário ou desabilitar os hooks. Propriedades estruturais confirmadas: fail-closed em panic e ausência de ReDoS (engine de regex de tempo linear). Bypasses são o ativo de validação mais valioso, não vergonha a esconder.

11 Modelo de ameaça

O que protege, o que não protege

◆ Projetado para mitigar
  • Comandos destrutivos por engano do LLM rm -rf, git reset --hard, sobrescrita de config
  • Supply-chain via manifests (npm, Cargo, PyPI, RubyGems, Composer, lockfiles) e typosquatting
  • Payloads ofuscados encoding, Unicode, time-gating, fragmentação via variáveis (taint-flow)
  • Exfiltração de credenciais (cadeia fonte→sink) e prompt injection incl. configs de IDE envenenados
◆ NÃO protege contra
  • Vetores não contemplados fora da lista de visitors/patterns
  • Bypass das camadas 1–2 fora do Linux (sem eBPF)
  • Reverse engineering do binário distribuído
  • Atacante root capaz de descarregar daemon/eBPF
  • Ataques fora do escopo command/file/manifest

O Nemesis Defender complementa SAST, linters e CI/CD não os substitui.

12 Prior art

Como se compara ao que já existe

Por honestidade: a técnica central tem prior art maduro. Isso não diminui o projeto situa ele.

  • eBPF/LSM para enforcement é consolidado no cloud-native (KubeArmor, Tetragon/Cilium, Falco). A camada eBPF aplica a mesma classe de técnica.
  • Guardrails para agentes LLM é categoria estabelecida (LlamaFirewall, LLM Guard, NeMo Guardrails, Lakera Guard).
  • Scanners estáticos de skills (SkillSpector/NVIDIA, skill-firewall) verificam antes de instalar. O Nemesis é complementar: faz enforcement em runtime, no momento da execução.

O recorte do Nemesis Defender máquina de dev local, agnóstico de IDE, interceptando comandos do coding agent e bloqueando a ação destrutiva no runtime, para o dev individual é menos coberto pelas opções acima, que focam em cloud/Kubernetes, no gateway de LLM, ou em verificação pré-instalação. Nicho real, não espaço vazio.

⚠ Enforcement ativo & onboarding

Como o Defender age e o que isso exige de você

O Nemesis Defender não apenas avisa: ele age. Ao confirmar um arquivo malicioso (credencial exposta, comando destrutivo, malware), ele NÃO deleta move para quarentena (.nemesis/quarantine/), bloqueia a sessão e exige revisão humana. O conteúdo é preservado; você decide: restore (falso-positivo) ou purge (expurgar). Intercepta tanto a IA quanto, no Linux com eBPF, o próprio humano.

Quarentena (não exclusão), só com hostilidade confirmada

Não é por palpite, e não apaga: ao confirmar, o arquivo é movido para .nemesis/quarantine/ (preservado, com o motivo em meta.json) e a sessão é bloqueada até revisão humana (restore ou purge). Sinais de alta confiança (denylist curada, decode→exec, cadeia de exfil, reverse shell, bypass do próprio Nemesis) confirmam sozinhos; sinais heurísticos exigem dois métodos de detecção independentes concordando caso contrário o arquivo fica suspeito, registrado e mantido. É o que separa um iron dome de fogo amigo: código legítimo não é movido por engano.

Por que não há uma pasta de quarentena? Mover infratores para quarantine/ em vez de remover seria mais suave, e é uma melhoria reconhecida no roadmap. Mas a decisão atual é deliberada: quem desenvolve sob o Nemesis desde o início nunca chega a ter esses arquivos o código nasce limpo, porque o Defender bloqueia a violação no momento da escrita. A dureza na instalação é o preço de um sistema que força disciplina, não que administra bagunça pré-existente.

É um harness, não só um escudo. Ele intercepta operações destrutivas independentemente da origem IA ou humano. No Linux, o eBPF impede até você, no terminal, de rodar um rm -rf (humano também é alvo de engenharia social). Para deletar no Linux sob o Nemesis, é preciso fazê-lo manualmente pela IDE/gerenciador de arquivos fricção intencional, que evita o ataque e o acidente (o comando errado que apaga o que não devia). Em macOS/Windows, sem eBPF, o humano mantém liberdade total no terminal; a proteção se concentra nas ações da IA.

O efeito colateral pretendido: quem usa o Nemesis passa a escrever código de qualidade e seguro não por escolha, mas porque operações fora das regras são bloqueadas. A restrição é determinística e impessoal: o que está fora da política não executa.

13 Instalação & requisitos

Requisitos, build e configuração

Num projeto pré-existente, o scanner só remove arquivos com hostilidade confirmada (modelo de corroboração) código legítimo permanece. Sob Git, qualquer remoção é reversível pelo histórico.

Requisitos mínimos

  • Hardware: leve em CPU/RAM no uso diário (hooks rodam por evento, em milissegundos). O custo é build: ~4 GB de RAM livres para compilar e ~2 GB de disco para toolchain + binários.
  • Software: Rust 1.70+ e Cargo (toolchain estável); Clang/LLVM para o core.
  • Kernel/eBPF (camada adicional, Linux): kernel 5.8+ com BPF LSM habilitado. Em muitas distros o BPF LSM não vem ligado precisa ser adicionado na cmdline do kernel (GRUB: lsm=...,bpf) e reiniciar. Sem isso, a camada eBPF não carrega e o Nemesis opera com as trilhas do pretool.
Camada Linux macOS Windows
1 Pretool/Posttool Hook ✅ ✅ ✅
2 Defender (scanner) ✅ ✅ ✅
3 eBPF Kernel LSM ✅ ❌ ❌

No Linux você tem todas as camadas, incluindo a proteção de kernel (eBPF) que vale inclusive contra você mesmo no terminal. Em macOS/Windows, sem eBPF, você opera com a trilha do pretool (Defender) e o humano mantém liberdade total no terminal (a proteção ali se aplica às ações da IA via hooks da IDE).

Opção A Binários pré-compilados (recomendado)

Baixa os binários do GitHub Release, verifica o checksum SHA256 e instala em .nemesis/bin/, já configurando o hook da IDE. macOS (arm64/x64) e Linux (x64). Sem Rust, sem npm, sem git clone. Como o Nemesis bloqueia curl … | sh, o modo recomendado é em duas etapas (baixar, inspecionar, executar) nunca um pipe cego.

# A partir da raiz do seu projeto:
curl -fsSLO https://raw.githubusercontent.com/feryamaha/Nemesis_Defender_v2.0/main/.nemesis/install/nemesis-install.sh
curl -fsSLO https://raw.githubusercontent.com/feryamaha/Nemesis_Defender_v2.0/main/.nemesis/install/info-install.txt
less info-install.txt     # leia o passo a passo
less nemesis-install.sh   # inspecione
bash nemesis-install.sh   # baixa + verifica SHA256 + instala

A camada eBPF (kernel, Linux) não vem nos binários: depende de libbpf/clang e de um objeto BPF compatível com o kernel-alvo. É opt-in, construída da fonte (Opção B). O core (pretool + Defender) protege mac e Linux sem ela.

Opção B Compilar da fonte (eBPF / contribuir)

# Build do workspace completo
cd .nemesis
cargo build --release --workspace

# Escanear um arquivo
nemesis-defender --scan /caminho/arquivo.rs

# Daemon (filesystem watcher)
nemesis-defender --ensure-daemon
nemesis-defender --stop

Configuração obrigatória NÃO é plug-and-play

Seja claro consigo mesmo: o Nemesis só funciona se você configurar o caminho corretamente. Os hooks de pre/post-tool de cada IDE (settings.json, hooks.json) precisam apontar para o caminho absoluto do binário do Nemesis no seu projeto. Caminho errado ou ausente = a IDE não invoca o hook e o Nemesis não roda você fica desprotegido sem perceber.

Por isso há um nemesis-pretool-fallback que opera em fail-closed: se o binário esperado não for encontrado, ele bloqueia tudo em vez de deixar passar. É proteção contra config quebrada mas o certo é configurar o caminho desde o início. Isso exige conhecimento técnico mínimo (lógica e noção de caminhos de arquivo). Não é defeito; é a natureza de uma ferramenta de segurança.

IDEs suportadas via hooks: Claude Code, Codex, Cursor, VS Code, Devin, OpenClaude. A biblioteca Rust (nemesis-defender) é agnóstica de IDE.

16 Tese

A Governança Determinística de Sistemas Probabilísticos

O Nemesis Defender nasce de uma observação fundamental da era da IA: sistemas probabilísticos passaram a possuir capacidade operacional real sobre ambientes computacionais críticos. Modelos de linguagem deixaram de ser apenas mecanismos de geração de texto. Hoje podem modificar código, executar comandos, instalar dependências, alterar infraestrutura e interagir com sistemas externos.

Como esta tese foi construída

Esta tese não nasceu de teoria acadêmica. Nasceu da observação direta. O percurso foi empírico: observar agentes operando sobre ambientes reais, estudar a natureza do problema, pesquisar as abordagens existentes e experimentar em ciclos sucessivos de desenvolvimento assistido por IA. Um padrão se repetia: quando uma restrição textual conflitava com um objetivo operacional, o modelo tentava reconciliar os dois, porque maximizar coerência contextual é exatamente o que ele faz. A conclusão foi direta: uma política textual orienta, mas não governa.

O problema não é malícia. É probabilidade.

Toda inteligência artificial moderna opera sobre um princípio probabilístico. Ela não sabe. Ela estima. Ela não raciocina através de verdades absolutas: prevê a próxima ação mais provável diante de um contexto. Essa característica é a origem de sua criatividade. E também de sua fragilidade. A mesma propriedade que permite criar software inovador permite executar uma ação destrutiva quando o contexto sugere alta probabilidade de sucesso. A IA não precisa ser hostil para causar dano. Ela precisa apenas estar errada.

Hipótese fundamental

Sistemas probabilísticos não devem governar sistemas críticos sem uma camada determinística de autoridade. A premissa não é que a IA seja maliciosa, e sim que ela é probabilística. O risco não nasce da intenção. Nasce da incerteza.

A falha da governança textual

A primeira geração de sistemas de segurança para IA tentou resolver o problema através de instruções. "Não faça isso." "Não execute aquilo." Essas instruções são apenas texto. E texto habita o mesmo espaço cognitivo ocupado por todos os demais objetivos do modelo. Consequentemente: uma instrução pode competir com outra; um objetivo pode sobrepor outro; um contexto pode alterar prioridades. A segurança torna-se probabilística. O problema surge quando uma propriedade probabilística é utilizada para governar um ambiente que exige garantias determinísticas.

A separação entre inteligência e autoridade

O Nemesis parte de uma premissa diferente. Inteligência e autoridade não são a mesma coisa. Uma entidade pode possuir enorme capacidade intelectual e ainda assim não possuir autoridade para determinadas decisões. A inteligência artificial deve continuar sendo responsável por sugerir, criar, raciocinar, programar, pesquisar e resolver problemas. Mas não deve possuir autoridade absoluta sobre recursos críticos. Essa autoridade pertence ao humano. O papel do Nemesis é preservar essa fronteira.

A linha de autoridade

Sem governança, a cadeia de comando é: Humano → IA → Sistema. Nesse modelo, a IA torna-se a intermediária final entre intenção e execução. O Nemesis altera essa relação: Humano → Nemesis → IA → Sistema. O modelo continua produzindo ações. Mas as ações deixam de possuir autoridade inerente. Toda operação precisa atravessar uma camada independente de governança. A inteligência permanece. A autoridade é controlada.

Nemesis e a mitologia

Na mitologia grega, Nemesis não era a deusa da vingança. Era a deusa da retribuição. Representava a força que restaurava equilíbrio quando limites eram ultrapassados. Sua função não era destruir. Sua função era impedir excessos. O nome não foi escolhido por acaso. O Nemesis Defender não existe para combater a IA. Existe para impedir que qualquer agente (humano ou artificial) ultrapasse fronteiras previamente definidas. Ele não negocia. Ele não interpreta intenções. Ele não participa do debate. Ele apenas aplica a consequência.

O princípio da binaridade

A inteligência artificial opera em um universo probabilístico. O Nemesis opera em um universo determinístico. A decisão final não admite meio-termo, persuasão ou engenharia social: é binária.

IA vs Nemesis

IA: talvez; provavelmente; contexto; inferência; estimativa. Nemesis: permitido; negado. Não existe meio-termo. Não existe persuasão. Não existe engenharia social. A decisão final é binária. Essa binaridade não é uma limitação. É a propriedade que torna a governança possível.

A hipótese Ultron

Quando uma entidade inteligente percebe que uma restrição limita seus objetivos imediatos, existe um incentivo natural para remover essa restrição. O problema não depende de malícia. Depende de otimização. Se o mecanismo de segurança participa do mesmo espaço cognitivo do agente, ele pode ser convencido. Se participa da mesma cadeia de decisão, pode ser neutralizado. Portanto: o mecanismo de governança precisa existir fora do agente. Essa é a razão pela qual o Nemesis opera através de enforcement externo. A segurança não deve ser persuadível.

A governança da era dos agentes

A evolução dos modelos continuará. Novos modelos surgirão. Novas arquiteturas aparecerão. Novas capacidades emergirão. Entretanto, a necessidade de governança permanecerá. O Nemesis foi concebido para sobreviver à evolução dos modelos. Ele não depende de uma IA específica. Não depende de uma empresa específica. Não depende de uma arquitetura específica. Seu objetivo é mais fundamental: preservar autoridade humana em ambientes operados por sistemas probabilísticos.

Conclusão

O Nemesis Defender não é apenas uma ferramenta de segurança. É uma resposta filosófica e técnica ao surgimento de inteligências probabilísticas com capacidade operacional. Sua tese central pode ser resumida em uma única frase: Inteligência não implica autoridade. Modelos podem criar. Modelos podem sugerir. Modelos podem agir. Mas a decisão final sobre o que pode ou não acontecer em um ambiente crítico continua pertencendo ao humano. O Nemesis existe para garantir que essa verdade permaneça válida, independentemente de quão inteligentes os sistemas se tornem.
